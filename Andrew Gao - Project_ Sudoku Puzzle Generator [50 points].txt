My Java program codes a completed Sudoku puzzle. It generates a 9×9 grid in which each row, column, and 3×3 subgrid contains the digits 1 through 9 without repetition. The program creates a 9×9 2D array named Puzzle. This array is then looped through starting from left to right and top to bottom. Each loop then goes through a series of checks to determine the possible values and is then passed to another check to determine if the row needs to be reset. If all checks are passed, then a number from the possible values is chosen and the program moves forward to the next iteration. Once all iterations are complete, it is then printed to display a completed Sudoku puzzle.


To obtain possible values for each element to be added to the ArrayList numGen, it is required that the program loops through all possible values, those being 1 to 9, and uses the methods rowCheck, colCheck, and areaCheck to determine if an element in the 2D array Puzzle in the same row, column, and 3×3 subgrid is already occupied by the value. Each method has the same parameters, those being the 2D array Puzzle, the value currently being accessed, and the location of the element, which includes both the row and column values, along with the same return type of boolean. The methods rowCheck and colCheck both initialize an ArrayList named rowValues and colValues, respectively. They go through the 2D array Puzzle to add the present values in the row for rowCheck by adjusting the column value, and in the column for colCheck by adjusting the row value. Once all values of the row or column are added to rowValues or colValues, it is then checked using the ArrayList method contains to determine if the value being accessed is present. If the value exists within the ArrayList, then it returns true, and false if not.


The method areaCheck works differently. Although it has the same parameters and return type, it does not use an ArrayList to check for values. Instead, it finds the location of the top-left element of the 3×3 subgrid by determining the row and column it would occupy, referred to as localRow and localCol. To find these values, the method takes the row or column given by the parameter and subtracts it by the row or column mod 3. After localRow and localCol are found, the method loops through a nested for loop with conditions being localRow and localCol plus 3. It loops through each corresponding element in the 2D array Puzzle to determine if the value is already present. If the value exists, then the method returns true, and false if not.


If the value passes rowCheck, colCheck, and areaCheck, then the value is added to the ArrayList numGen. If numGen is empty, meaning there is no possible value that passes all three checks, then the program resets all values in the row to 0 and sets col to -1 to restart the row. If there are values present in numGen, then using random number generation, it assigns the element of Puzzle to one of the values in numGen. Once the nested loop completes, all elements of the 2D array Puzzle are assigned a value from 1 to 9.


During the process of creating the program, I came across several challenges that forced changes to some methods and the addition of more checks. When I was initially creating areaCheck, I began using a switch case and started coding for every possible position in the 3×3 subgrid that it could occupy. However, several issues arose while making it. The first was its length: all possible index values for rows and columns had to have a specific case value. More issues arose when implementing the switch case, as although it was using the row as the switch case, the column also had to be adjusted to get all values in the 3×3 grid. This caused an even greater increase in length, as in each case there had to be several more if statements to determine how many times it would need to adjust the loop and how many times the row and column had to be adjusted. It added all the values of the 3×3 grid to an ArrayList, similar to colCheck and rowCheck, and used the ArrayList to determine if the value was present. However, due to issues with the logic in the switch case, the length of the method, and the inefficiency of the overall process, it was deemed better to find a simpler solution.


Another issue arose with how I was determining the values that were being assigned to each element. Initially, I was randomly generating numbers from 1 to 9. However, it became clear that this would slow the program down, as the same number could go through all the checks multiple times, and it was not efficient. By instead looping through the possible digits, 1 to 9, I was able to reduce repetition and simply add the possible values to an ArrayList. The last and most urgent issue that was solved was that even if nothing went wrong with the checks and values assigned, there was still a possibility that for one or multiple elements, no values from 1 to 9 would pass all three checks. This is due to values that passed the checks being already assigned earlier in the row. To fix this issue, the available numbers in the ArrayList were checked, and if it was empty, the entire row was redone. Overall, the project was an interesting challenge, and I enjoyed the process of making it.
